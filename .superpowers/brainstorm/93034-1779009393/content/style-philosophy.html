What styling philosophy for `KeyBindingsView`?

This frames everything else in Phase 2 — recording UX, search, conflict display, theming. Pick the closest match; we can refine.

A

Native macOS Preferences look

SwiftUI Form + Section per context, system styling, indistinguishable from Xcode's "Key Bindings" pane or Settings → Keyboard → Shortcuts. Inherits OS dark/light + system fonts. Adopters who want a different look style their own via the headless KeyBindingsTable.

Pros: Zero theme work, looks "right" by default · Cons: Less distinctive, harder to make denser

B

Custom-styled, denser (Zed / VS Code feel)

Hand-rolled List rows, monospace shortcut display chips, search bar at top, conflict badges inline. Looks more like a power-user shortcut manager than a settings pane.

Pros: Distinctive, denser, scans faster · Cons: More theming code to maintain; opinionated style may clash with some adopters' apps

C

Two flavours: .style(.native) / .style(.dense)

One view, two opinionated presets selectable via a modifier. Default to .native. Adopters pick the preset that fits.

Pros: Best of both — drop-in for typical apps, dense option for power tools · Cons: ~2× rendering code; presets always lag behind real custom design

Either way, the headless KeyBindingsTable from Phase 1 stays the escape hatch — adopters who want a fully custom UI render their own from that data type.
